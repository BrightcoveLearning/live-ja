    ライブモジュールガイドラインおよびベストプラクティス

    Video Cloud Liveでライブイベントを簡単に設定し、Web、iOSおよびAndroid機器に多重ビットレートストリームを配信することができます。このトピックでは、高品質で安定したライブストリーミングエクスペリエンスの提供に役立つベストプラクティスおよび推奨事項について概説します。
    注：ライブストリーミングのサポートをご利用いただけるのは、Video Cloud ProおよびVideo Cloud Enterpriseをご利用のパブリッシャーのみです。

    ライブイベントの放送をすぐに開始したい場合は、次の項目をチェックしてください。

    • ステップバイステップ:ライブモジュールと三ストリームWirecastの使用
    • ステップバイステップ:オープンブロードキャスタソフトウェア (OBS) でのライブモジュールの使用
    ライブストリーミング動画を配信するための Brightcove で利用可能なすべてのオプションの一覧については、「ライブストリームの配信」を参照してください。

    認定エンコーダー

    サポートされているエンコーダの一覧については、「ライブイベントでサポートされるエンコーダ」を参照してください。その他のハードウェアおよびソフトウェアベースのエンコーディングツールもLiveモジュールで使用できる場合がありますが、それらのツールの使用はサポートされません。特定のツールについてご質問がある場合は、ブライトコーブのサポートまでご連絡ください。

    入力ストリームのガイドライン

    視聴者に最高のユーザー視聴エクスペリエンスを提供する唯一確実な方法は、高品質で安定した入力ストリームを提供することです。品質の高い入力ストリームを提供することで、特定の場所から一貫して利用可能な最大帯域幅で最高の動画品質を実現できます。

    入力制限

    最高品質で一貫したストリーミング体験を実現するために、ブライトコーブライブでは入力ストリーム設定を以下の項目に制限しています。

    • 最大1920x1080
    • 最大30fps、これは典型的です。ブライトコーブは最大 60 fps をサポートしますが、上限を増やすには、ブライトコーブサポートに連絡する必要があります。60fps を使用する場合、ブライトコーブではビットレートを大幅に上げて、一定のフレームレートを使用することをお勧めします。
    • 10 Mbps未満
    • 一定ビットレート（CBR）は、問題の発生可能性を大幅に低減
    • オーディオサンプリングレート:44.1khz と 48khz は、推奨されるサンプルオーディオレートです。
    • キーフレームレートまたは GOP (ピクチャのグループ) の位置合わせ:
      1. キーフレームは、入力と出力の両方で常に 2 秒ごとに発生します。つまり、ストリーム自体の 2 秒ごとにエンコーダから Brightcove にキーフレームが送信されます。このプロセスはさまざまな方法で定義できますが、キーフレームレートが最も一般的です。
      2. これは、異なるエンコーダによって異なる方法で計算することができます。例は次のとおりです。
        • Wirecast は通過するフレームの量を使用するため、30 fps のビデオの場合、設定は 60 になります。
        • エレメントエンコーダは、正しい設定が '2' になるように、秒を使用します。
        • 60 FPS ビデオは、この設定がフレームでカウントされる場合にのみ変更されます。この場合、120 フレームごとに 2 秒になります。
    • [ キーフレームの位置合わせ ]、[ 同期GOP ]、[ キーフレームを位置合わせ]、または [キーフレームを位置合わせ ] などのオプションがある場合は、キーフレームが揃っていることを確認してください。キーフレームが揃っていないと、HLS セグメンテーションで同期の問題が発生します。

    帯域幅のテスト

    入力ストリームの適切な設定を見つける第一歩は、現場で利用可能な帯域幅を特定することです。次のようなツールが役立ちます。

    • SpeedOf.Me（http://speedof.me）- HTTP接続に利用可能な帯域幅合計を特定することは、適切な第一歩です。しかし、入力フィードはHTTPではなく、RTMPでLiveモジュールにストリーミングされるため、RTMP接続で利用可能な実際の帯域幅は相当少なくなります。
    • Speedtest ( http://www.speedtest.net )-現在のアップロード速度とダウンロード速度を決定するオンラインツール。

    入力帯域幅

    視聴者に最高のユーザー視聴エクスペリエンスを提供する唯一確実な方法は、高品質で安定した入力ストリームを提供することです。品質の高い入力ストリームを提供することで、特定の場所から一貫して利用可能な最大帯域幅で最高の動画品質を実現できます。

    • 最小入力帯域幅:2.5メガビット/秒
    • 最大入力帯域幅:10 mbps
    注：最高のユーザー視聴エクスペリエンスを提供するために、出力レンディションをエンコーダーストリームの品質に合わせます。

    エンコーダー機能の決定

    ライブストリームをエンコードし、Liveモジュールに送信するソフトウェアおよびハードウェアの機能について理解することも重要です。高品質な1080p入力ストリームの送信に十分なビットレートが確保できでも、ハードウェアがエンコードを行うにはリアルタイムよりも速い速度が必要です。一部のエンコーディングツールでは、合計CPU使用率および使用帯域幅に関する情報が表示されます。たとえば、Telestream Wirecast では、出力統計がページの上部に表示されます。この情報は、特定のハードウェア上で可能な最も安定した、最高品質のストリームを判断するときに役立ちます。

    Wirecastで確認する項目は次のとおりです。

    • CPU使用率が80%未満であること
    • データレートが目標ビットレートに近いこと
    • FPSが入力ストリーム設定のレートであること

    イベントの準備

    イベントの制作にLiveモジュールを使用する前に、エンドツーエンドのチェーン全体をテストすることが成功の確保には不可欠です。主なテスト領域は次の2つがあります。

    • ビデオ制作ワークフロー
    • Live モジュールへの接続に使用されるツール

    ロケーションを追加する、またはハードウェアまたはソフトウェア構成を変更するたびに、新たにテストを行う必要があります。

    制作

    生成されて、エンコーディングハードウェアまたはソフトウェアに送信されるライブフィードは、Webcamのような単純なものからマルチカメラの編集済みライブブロードキャストのような複雑なものに至るまで多岐にわたります。いずれの場合も、入力動画フィードがイベントを通じて一貫している必要があります。カメラアングルやシステムを切り替えた場合、同じ解像度および音声チャンネル数になるよう確認してください。

    Liveモジュールへの接続

    Liveモジュールへの接続はイベント制作で使用するものと同じネットワークで同じ入力ライブフィードを使用し、同じエンコーディングソフトウェアを通じてテストする必要があります。イベント制作を可能な限り忠実に模倣することが目標です。

    イベント当日

    放送予定の遅くとも10分前にはライブストリームを開始するよう予定を立てる必要があります。30分前に開始すると、調整や問題解決のために時間の余裕が得られます。

    ライブストリーミング中の問題

    ライブストリームのテスト、入力ストリームの理想的な設定の特定、イベント会場の準備により、エラーが発生する可能性を大幅に低減できます。しかし、起こりうる問題に対処できるよう常に準備しておくことが重要です。

    切断

    ネットワークの停止、エンコーダーの故障または停電などはすべて、Liveモジュールへの接続が失われる原因となります。ライブイベントには [再接続時間 ] があります。これは、ストリームが再接続されるまで待機する時間 (分) です。接続が切断されると、Live モジュールは選択した再接続時間後にを切断します。イベントの作成時に [詳細オプション ] を展開して、再接続時間を設定します。

    Liveモジュールでは、この再接続時間内に切断および再接続を回数制限なく受付けます。新しいストリームを開始する必要はありません。ストリームが再接続されると、最も新しいフレームから再生が再開されます。ストリーミングが再接続されることなくイベントが終了した場合、プレーヤーはストリーミングが見つからないという警告画面に戻ります。

    注：切断および再接続時には、同じストリーム設定にしておく必要があります。音声チャンネル数、解像度またはコーデック設定などを変更すると、予測できない動作の原因となります。

    待機時間

    ストリーム待機時間、または視聴者が視聴しているイベントと実際のイベントとの間の時間差は、次の3箇所で発生します。

    • オンサイトのエンコーダ
    • クラウドトランスコーディング
    • CDN配信

    現場のエンコーダーおよびLiveモジュールのクラウドトランスコーディングシステムでは、合計1.5～5秒の待機時間が発生します。CDNによっては、さらに15～90秒の待機時間が発生します。一般的に、15～90秒の待機時間はライブフィードでは正常であり、Liveモジュールがライブイベントの数百万の同時視聴者にサービスを提供するために必要であると見なされます。

    ライブストリームの問題をトラブルシューティングする

    問題 解明
    動画がフリーズし、音声のみ再生される エンコーダーへの入力フィードを確認し、エンコーダーが適切に接続されており、動画がエンコードされていることを確認します。切断が発生した場合は、再接続の前にストリーム設定が同じであることを確認します。
    プレーヤーでストリームがフリーズしたように見える Liveモジュールで入力エンコーダーからのデータ受信が停止している可能性があります。入力ストリームがアクティブで接続されており、リアルタイムよりも高速にデータを送信していることを確認します。
    再生が途切れがち 入力ストリームのビットレートをサポートするために十分以上の帯域幅があることを確認します。一般的に、再生の途切れは制作サイドに何らかの問題があるか、帯域幅が不十分であることを示しています。
    画面に何も表示されない 十分な帯域幅があることを確認します。切断が発生した場合は、再接続の前に設定が変更されていないことを確認します。
    予想される待機時間よりも長い 利用可能な帯域幅を確認し、データがリアルタイムでLiveモジュールに送信されていることを確認します。ストリームの配信まで、最大10分程度かかる場合があります。この間はストリーミングがキャッシュされ、待機時間が最も長くなります。配信の開始後は、待機時間が徐々に短くなり、30～90秒で安定します。
    アカウントに基づいて割り当てられた冗長ストリームをすでに使用しました。 アカウントの制限にカウントする冗長グループがある場合があります。冗長ストリームを含むイベントが完了またはキャンセルされると、ライブモジュールに表示されなくなります。ただし、このようなイベントは Live API を使用して一覧表示および削除できます。ライブ API を参照してください。冗長グループ。
     

    さらにサポートが必要な場合

    お客様のライブイベントが機能するよう、さらにサポートが必要な場合は、弊社までお問い合わせください。なるべく早くご返信ができるよう、問題解決のためにブライトコーブサポートチームが必要とする情報を以下にリストアップしています。

    • ストリームに見受けられる特定の兆候。たとえば、一部が再生できない、スムーズに再生できないなど
    • 問題のストリームが過去に正しく機能していたか
    • エンコーダーで使用しているエントリーポイントURL
    • 使用しているエンコーディングソフトウェアおよびハードウェア
    • ライブイベントを公開したプレーヤーのURL
    • Video Cloud Studio内のライブアセットの動画ID
    • エンコーダーから公開ポイントホストへのルートトラッキングの結果
